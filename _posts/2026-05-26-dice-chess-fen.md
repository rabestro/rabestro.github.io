When building a game engine, one of the most critical components is how you represent the game state. In traditional chess, this is a solved problem thanks to **FEN (Forsyth-Edwards Notation)**. FEN allows you to describe any board position using a simple string.

But what happens when you introduce dice to the game?

In this post, we'll explore why standard FEN wasn't enough for the Dice Chess Engine, our early struggles, and how we arrived at a robust, backward-compatible solution we call **DFEN**.

## What is FEN?

Standard FEN consists of 6 fields separated by spaces:
1. **Piece placement**: Where the pieces are on the board (e.g., `rnbqkbnr/pppppppp/...`).
2. **Active color**: Whose turn it is (`w` or `b`).
3. **Castling rights**: Who can still castle (`KQkq` or `-`).
4. **En passant targets**: The square behind a pawn that just moved two squares.
5. **Halfmove clock**: The number of halfmoves since the last pawn advance or capture (used for the 50-move rule).
6. **Fullmove number**: The number of full turns completed.

An example of the starting position:
`rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1`

This works beautifully for classic chess. But Dice Chess is different.

## The Dice Chess Challenge

In standard chess, a turn is exactly one move. In Dice Chess, a turn is much more dynamic:
1. You roll the dice.
2. You receive a "dice pool" (e.g., `2`, `3`, `6`).
3. You can make up to 3 micro-moves, consuming a die from the pool for each move.

Standard FEN has absolutely no concept of a "dice pool". If we only use standard FEN, we know the board state, but we don't know:
- Which dice were rolled?
- Which dice have already been used in this turn?
- What dice are left for the remaining micro-moves?

## Early Attempts & Pitfalls

Initially, we tried to track the dice pool separately from the FEN string. The UI would send the FEN to the engine, and then in a separate field or WebSocket message, it would send the dice.

This quickly became a synchronization nightmare.
- **Race Conditions**: The UI would update the board, but the engine might process the FEN before receiving the updated dice pool.
- **Testing Difficulties**: Writing automated tests meant setting up complex state objects instead of simply passing a string.
- **Stateless Constraints**: HTTP requests and simple APIs couldn't rely on a single source of truth for the game state.

## Putting it to the Test (100 Games & Bug #209)

The breaking point came when we started playing intensely against the engine bot. After playing nearly 100 games, we discovered subtle edge cases where the bot would generate invalid moves or misinterpret its turn (such as [Issue #209](https://github.com/rabestro/dicechess-engine-scala/issues/209)). 

Debugging these issues was incredibly difficult without a unified string representation of the *entire* state. We needed a way to snapshot the exact moment a bug occurred—including the exact dice available—so we could write a test case for it.

## Our Solution: DFEN

We realized the game state *must* be a single, unified string. So, we extended FEN by simply adding a **7th field**: the dice pool.

For example, if the active player has a 2, 3, and 6 available, the DFEN looks like this:
`rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1 236`

When a die is consumed (e.g., the `2` is used), the next DFEN generated by the engine is:
`rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1 36`

And when the turn is over, the field is either omitted or left empty, making it perfectly compatible with the start of a new turn before dice are rolled.

## The Beauty of Backward Compatibility

You might wonder: *Didn't adding a 7th field break all the existing FEN parsers?*

Surprisingly, no! And this is where defensive programming in our UI (`frontend-pwa`) saved the day. 

Most robust FEN parsers (including our TypeScript frontend) parse the string by splitting on whitespace:
```typescript
const parts = fen.split(/\s+/);
const piecePlacement = parts[0];
const activeColor = parts[1];
// ...
```
By simply appending the dice pool to the end, existing parsers that only care about the first 6 fields simply ignore the 7th field. It doesn't throw an error; it just gracefully processes the standard chess data. 

The visual dice manager in the UI can then optionally check for `parts[6]` to render the remaining dice.

## Conclusion

Sometimes the best solutions are the simplest. By extending FEN to DFEN, we eliminated state synchronization bugs, made our engine infinitely easier to test, and maintained backward compatibility with our frontend tools. 

If you're building a chess variant, don't be afraid to extend the standards to fit your needs—just make sure you do it at the end of the string!
